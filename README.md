# Подход к реализации системы мониторинга блогов

## 1. Архитектура системы

### 1.1. Структура БД

- **`resources`** - информация о типах ресурсов
  - `name`, `driver` - имя и драйвер для вызова API
- **`blogs`** - блоги, находящиеся на мониторинге
  - `resource_id`, `identifier`, `check_interval`, `last_sync_at`, и т.д.
- **`posts`** - посты из блогов
  - `blog_id`, `identifier`, `title`, `content`, `rating`, `reactions`
- **`notifications`** - журнал уведомлений о новых постах
  - `blog_id`, `report` (JSON-массив новых постов)

### 1.2. Ключевые компоненты

- **`app\Services\BlogApi\Interfaces\BlogApiAdapterInterface.php`** - интерфейс для работы с API разных ресурсов
- **`app\Services\BlogApi\BlogApiAdapterFactory.php`** - фабрика для выбора нужного адаптера по `driver`
- **`app\Services\BlogSyncService.php`** - основной сервис синхронизации
- **`app\Jobs\SyncBlogJob.php`** - асинхронная задача для обработки блогов
- **`app\Console\Commands\RunBlogSyncCommand.php`** - команда для запуска синхронизации
- **`app\Http\Controllers\Api\BlogController.php`** - API-контроллер для добавления блогов на мониторинг пользователем

## 3. Паттерны проектирования, которые я использовал

- **Strategy** - для выбора нужного API-адаптера в зависимости от ресурса.
- **Factory** - централизованное создание адаптеров по драйверу.
- **Service Layer** - вынос всей логики синхронизации в отдельный класс.

### 3.1 Паттерны, которые я мог применить, но отказался:

- **Repository** - не использовался, так как Eloquent уже предоставляет достаточный уровень абстракции.
- **Decorator** - в теории мог использоваться для динамического изменения адаптеров, но здесь достаточно интерфейсов и наследования.

---

## 4. Обоснование выбора инструментов и особенности реализации

- **Queues & Jobs** - асинхронная обработка, чтобы не блокировать основной поток при десятках тысяч блогов.
- **FormRequest** - валидация входных данных с удобными сообщениями.

> Остается только **Schedule** - для запуска команд по расписанию, но в тестовом варианте я решил его не реализовывать.

- **Транзакции** - для обеспечения целостности данных при синхронизации.
- **Chunk-запросы** - для обработки большого количества блогов без переполнения памяти.
- **Upsert** - вставка постов по уникальному ключу, в данном случае он будет лучше при большом количестве данных.
- **Casts** - касты для автоматического преобразования полей в моделях.
- **Seeders** - сидеры для заполнения базы моковыми данными.

---

В качестве СУБД использовал **SQLite** для простоты и скорости (легче развернуть, в том же Docker).  
Из-за использования SQLite некоторые запросы пришлось адаптировать:  
вместо функций типа `DATE_ADD` используется `datetime(last_sync_at, '+' || check_interval || ' seconds')` для работы с датами в `RunBlogSyncCommand`.

Ещё в теории проверку на интервал времени можно было сделать в базе с помощью `CHECK`, но, во-первых, это нагрузка на базу, а во-вторых, в теории это специальная обработка ошибок для пользователя.
